# Context Manager

## Purpose

Produce a **bounded reading list** — the minimum set of files Claude needs to load into context
before acting on the current feature. This avoids full-repo scans on large codebases (especially
multi-module Java/Spring microservices) and keeps token usage proportional to feature scope.

## Inputs

1. **Active spec** — `specs/features/<NNN>/SPEC.md` (required).
2. **Context docs** (read if they exist, skip gracefully if absent):
   - `docs/PROJECT_CONTEXT.md`
   - `docs/TECH_STACK.md`
   - `docs/ARCHITECTURE.md`
3. **Graph map** (optional accelerator):
   - `GRAPH_REPORT.md` at project root (produced by Graphify).
   - If present **and fresh**, use it to identify impacted modules.
   - If absent or stale, fall back to bounded heuristic scan.
4. **PLAN.md / TASKS.md** — to understand which task is active.

## Behavior

1. **Read context docs** (if they exist) to understand boundaries, modules, and communication patterns.
2. **Read the active spec** — extract impacted domains, endpoints, entities, events, and data stores.
3. **Check for GRAPH_REPORT.md:**
   - If present and not stale (mtime newer than or within 7 days of latest source change):
     extract the subgraph relevant to the impacted modules.
   - If absent or stale: note it and proceed with heuristic.
4. **Heuristic fallback** (when no graph map):
   - From the spec's declared changes (API, data model, messaging), infer which packages/directories
     are likely impacted.
   - Limit to at most **15 files** for initial context load.
5. **Output the reading list:**
   - Ordered by relevance (most impacted first).
   - Each entry: file path + one-line reason why it's relevant.
   - Clearly marked: "Read these before implementing. Skip everything else unless a finding points there."
6. **Cache the list** in `DECISIONS.md` of the active feature (under a `## Context reading list` section)
   so re-entry to the same feature doesn't re-derive it.

## Output format

```markdown
## Context reading list

Generated by /context-manager for spec NNN-feature-name.

| # | File | Reason |
|---|---|---|
| 1 | `src/main/java/.../OrderService.java` | Service containing the affected business logic |
| 2 | ... | ... |

**Graph map status:** fresh / stale (N days) / absent
**Fallback used:** yes / no
```

## Graceful degradation

- No `docs/PROJECT_CONTEXT.md` → skip; note "project context not available, scanning broader".
- No `GRAPH_REPORT.md` → use heuristic; note "Graphify map not available — consider running Graphify for better precision".
- No `PLAN.md` yet → use SPEC.md alone.

## What this skill does NOT do

- Does not run Graphify (that's `graphify-context`).
- Does not modify application code.
- Does not replace reading the actual files — it only decides *which* files to read.
- Does not block on missing context docs — always produces a best-effort list.
