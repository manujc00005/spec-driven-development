# SDD Onboard

## Purpose

Bring an existing codebase into the SDD workflow **without modifying its application code**. Produce
context documents that let Claude (and the engineer) understand the project's stack, architecture,
patterns, and constraints before writing any specs or plans.

## When to use

- Starting SDD in a project that already has code but no `docs/` context or `specs/` structure.
- After cloning a new-to-you repo and wanting Claude to "learn" it.
- As a prerequisite before `/spec-create` on a project Claude hasn't seen before.

## Inputs

The project root. The skill reads (never modifies) application files to detect patterns.

## Behavior

### Step 1: Stack detection (read-only)

Scan for build/config files and infer the tech stack:

| File | Infers |
|---|---|
| `pom.xml` / `mvnw` | Java + Maven (primary build tool) |
| `build.gradle` / `gradlew` | Java + Gradle |
| `package.json` | Node.js / TypeScript |
| `next.config.*` | Next.js |
| `schema.prisma` | Prisma ORM |
| `angular.json` | Angular |
| `Dockerfile` / `docker-compose.yml` | Containerized |
| `helm/` / `Chart.yaml` | Kubernetes + Helm |
| `application*.yml` / `application*.properties` | Spring Boot |
| `Cargo.toml` | Rust |

Report detected stack to user before proceeding.

### Step 2: Architecture scan (read-only)

- Identify top-level modules/packages from directory structure.
- Detect layering patterns (controllers → services → repositories, or similar).
- Detect messaging: Kafka/RabbitMQ config, listener/producer annotations.
- Detect external integrations: HTTP clients, Feign, WebClient.
- Detect data stores: JPA entities, schema files, Redis config.
- Detect security: Spring Security config, Keycloak, OAuth2 resource server.

### Step 3: Pattern detection (read-only)

- Naming conventions (packages, classes, DTOs, endpoints).
- Error handling pattern (global exception handler, Result types).
- Testing patterns (which frameworks, where tests live, test profiles).
- Config management (profiles, Vault, env vars).

### Step 4: Scaffold context docs (writes only to docs/ and specs/)

Generate the following files, pre-filled with detected information:

1. **`docs/PROJECT_CONTEXT.md`** — from `docs/_templates/PROJECT_CONTEXT.md`, filled with detected modules and stores.
2. **`docs/TECH_STACK.md`** — from `docs/_templates/TECH_STACK.md`, filled with detected versions and commands.
3. **`docs/ARCHITECTURE.md`** — from `docs/_templates/ARCHITECTURE.md`, filled with detected layers and dependencies.
4. **`specs/CONSTITUTION.md`** (skeleton, if it doesn't exist) — basic rules inferred from patterns.

Each generated file is marked with `<!-- Auto-generated by /sdd-onboard. Review and adjust. -->` at the
top so the user knows it needs human validation.

### Step 4b: Optional Graphify detection (read-only, non-mandatory)

If the project has Graphify installed, optionally detect and scaffold architecture guidance:

- Check if `.graphify/graph.json` or `.graphify/GRAPH_REPORT.md` exists.
- If found: optionally scaffold `docs/PROJECT_GRAPH.md` from `docs/_templates/PROJECT_GRAPH.md`.
- If not found: skip silently. (Graphify is optional, not required.)
- Never execute Graphify commands. Never install Graphify. Never auto-generate the graph.

Report to user: "Graphify detected. Scaffolded `docs/PROJECT_GRAPH.md`" or "(Graphify not installed — optional)".

**Note:** This is a convenience detection only. Projects work fully without Graphify.

### Step 5: Report to user

Print a summary:
- Detected stack.
- Detected architecture style.
- Files generated.
- Gaps that need manual input (e.g., "could not detect deployment topology — fill in ARCHITECTURE.md §Deployment").
- Suggested next step (usually `/project-init` to complete CONSTITUTION, or `/spec-create` to start a feature).

## Output

- 3-4 files created under `docs/` and optionally `specs/`.
- Summary report printed to conversation.
- **No application code modified. Ever.**

## Graceful degradation

- If the project has no recognizable build file: report "Could not auto-detect stack. Please fill in `docs/TECH_STACK.md` manually." and generate empty templates.
- If a context doc already exists: do NOT overwrite — report "docs/TECH_STACK.md already exists, skipping. Use `/spec-update` to modify it."
- If `specs/CONSTITUTION.md` already exists: skip — report "CONSTITUTION already present, not overwriting."

## What this skill does NOT do

- Does not modify application code (Java, TypeScript, config, SQL, etc.).
- Does not run builds, tests, or Graphify (Graphify is optional, run externally if desired).
- Does not delete or overwrite existing docs.
- Does not make architectural recommendations — it *documents* what exists.
- Does not create feature specs — that's `/spec-create`.
- Does not require Graphify to be installed. Works fully without it.
- Does not auto-install Graphify or modify shell profiles.
